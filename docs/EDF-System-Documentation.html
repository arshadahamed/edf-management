System Architecture

The EDF system follows a classic MVC-adjacent pattern: Express routes handle HTTP requests, SQLite provides persistence, and static HTML/JS files served from /public form the frontend. There is no build step for the frontend — all JS and CSS are authored directly.

Request Lifecycle

  • 1
    Browser → Express — HTTP request arrives at port 5001.
  • 2
    maintenanceGuard — If maintenance mode is on and the request is for a public page, redirect to /maintenance. API and admin routes pass through.
  • 3
    Static Files — express.static('public') serves HTML, CSS, JS, and uploaded files directly.
  • 4
    activityLogger — Logs the incoming API request (user, action, entity, IP) to system_logs.
  • 5
    userGuard — Verifies the JWT cookie is valid, the account is not blocked, and no force-logout has been issued. Updates last_seen timestamp.
  • 6
    permissionGuard — For mutation methods (POST/PUT/DELETE), checks the user has the required module permission or is admin/super_user.
  • 7
    Route Handler — The matching Express route executes the business logic, queries SQLite, and returns a JSON response.
  • 8
    Response → Browser — JSON response is consumed by the frontend JS, which updates the DOM without a page reload.

Authentication & Authorization

4.1 JWT Authentication Flow

Authentication uses JSON Web Tokens (JWT) stored in an httpOnly cookie. The httpOnly flag prevents JavaScript from reading the token, protecting against XSS-based token theft.

  • 1
    User submits username and password on the login page (POST /api/auth/login).
  • 2
    Server verifies the password with bcryptjs.compare() against the stored hash.
  • 3
    On success, a JWT is signed with JWT_SECRET and sent as an httpOnly cookie. The token contains: userId, username, role, iat (issued-at timestamp).
  • 4
    All subsequent API requests automatically include this cookie. The userGuard middleware verifies it on every /api/* request.
  • 5
    On logout (POST /api/auth/logout), the cookie is cleared server-side and a logout event is logged.
Force Logout: Admins can force-logout any user remotely. The server sets a force_logout_at timestamp on the user record. The userGuard then rejects any token minted before that timestamp, even if the JWT itself is still valid.

4.2 User Roles

Six roles exist with hierarchical privilege levels. admin and super_user bypass all module permission checks.

RoleLevelDescription
adminHighestFull system access. Can manage users, permissions, settings, and all modules. Bypasses all permission guards.
super_userVery HighFull operational access. Same as admin for data operations but cannot manage users/settings.
managerHighAccess determined by granted module permissions. Typically has write access to most modules.
editorMediumAccess determined by granted module permissions. Typically read/write on assigned areas.
viewerLowRead-only access to permitted modules. Cannot perform POST/PUT/DELETE operations.
userLowestMinimal access. Only permitted modules are accessible.

4.3 Module Permissions

Non-admin roles must be explicitly granted permission per module. Permissions only affect mutation operations (write). Read (GET) requests are not permission-gated.

Permission KeyControls Access To

beneficiariesCreating, editing, deleting beneficiary records and nested children
financeCreating donations, subscriptions, family payments, external donations
volunteersCreating and editing volunteer records
coursesCreating programs, batches, and managing applications
ramadanCreating distributions, assigning tokens, marking collection
reportsAccessing report exports and log details
settingsChanging maintenance mode, uploading logo/favicon
usersCreating and managing user accounts (admin panel)

Admin Dashboard — Module Guide

The admin dashboard is accessible at http://localhost:5001/dashboard after logging in. It is a Single Page Application (SPA) — all modules load within the same page without navigation. Click a sidebar item to switch modules.

Theme Toggle: The dashboard supports both Light and Dark themes. Click the theme icon in the top bar to switch. The preference is stored in localStorage.

7.1 — Overview Dashboard

The first page shown after login. Displays real-time KPIs and summary statistics fetched from GET /api/stats/stats.

What you see:

  • KPI Cards — Total members, beneficiaries, volunteers, programs, total donations collected, expenses.
  • Active Users Panel — Shows staff currently online (last seen within 90 seconds) with their name, role, and avatar.
  • Recent Transactions — Last 10 financial events (income and expenses).
  • Ramadan Stats — Current year's token distribution progress.

7.2 — Member Management

Manage all foundation members. Accessible via the Members sidebar link. Requires members permission for write operations.

Listing Members

  • Search by name, email, NIC, or phone using the search bar.
  • Filter by member type (Executive / Founder) and status (Active / Deactivated) using dropdowns.
  • Members are displayed in a table with: Name, Member Type, Phone, Monthly Subscription, Join Date, Status.

Adding a New Member

  • 1
    Click the + Add Member button (top right of the members table).
  • 2
    Fill in the required fields: Full Name, Member Type, Phone, Join Date, Monthly Subscription amount.
  • 3
    Optionally fill in: Email, NIC, Address, City, Gender, Age, Income, Marital Status, Blood Group, Occupation, Bank Details, Previous Balance.
  • 4
    Click Save Member. The system validates NIC uniqueness across all members.

Editing / Deleting a Member

  • Click the Edit (pencil) icon in the member row to open the edit form.
  • Click the Delete (trash) icon to delete. A confirmation prompt appears before deletion.
  • Member status can be changed between Active and Deactivated using the status dropdown in the edit form.

Member Relatives

  • Expand a member row by clicking it to reveal the Relatives section.
  • Add relatives (wife, son, daughter, other) with their name, age, NIC, occupation.
  • Relatives are stored in member_relatives and are shown only within the context of their parent member.

Payment History Modal (CRUD)

Click the history icon on any member row to open the Member History Modal. This modal has two tabs:

  • View History — Read-only table of all subscription months. Supports search by month, filter by year, sort by date/amount, and CSV export.
  • Manage Payments — Full CRUD interface for subscription records:
  • 1
    Add Month — Click + Add Month to expand the new-month form. Enter: Month, Expected Amount, Payment Date, Payment Method, Notes. The system prevents duplicate months for the same member.
  • 2
    Expand a Month Row — Click any accordion row to expand. The expanded view shows two sections: the Month Record (expected amount, method, notes) and Installments (individual payments).
  • 3
    Edit Month Record — Click Edit in the Month Record section to make fields editable. Change values and click Save.
  • 4
    Delete Month — Click Delete Month button. An inline confirm strip appears (row turns red). Click Yes, Delete to confirm or No to cancel.
  • 5
    Add Installment — Click + Add Installment to append a blank input row. Enter Amount, Date, Method, Notes and click ✓ to save.
  • 6
    Edit/Delete Installment — Each installment row has ✏ (edit inline) and ✕ (delete with confirm) buttons. Deleting recalculates the month's paid_amount automatically.
Validation: You cannot delete the last installment on a month. Delete the whole month instead.

7.3 — Family Registry

Track families receiving welfare aid. Each family has a head, address, member count, and poverty level classification.

  • Add Family — Click + Add Family. Fill in: Head Name, Address, Phone, Member Count, Poverty Level (Low / Medium / High).
  • Family Donations — Each family has a Donations sub-section. Add donations with: Amount, Donation Type (Cash / Dry Ration / Zakat), Description, and Date.
  • Filter by poverty level or search by head name.

7.4 — Beneficiaries

The most complex module. Manages aid applicants/recipients with detailed household data. Requires beneficiaries permission.

Sub-sections in the Sidebar:

  • All Records — Table of all beneficiary records with search, filter by status and category, and bulk import/export via Excel (XLSX).
  • Register New — Multi-field registration form with both male and female head details, income data, children data, and photo upload.
  • Categories — Manage beneficiary category labels (create/edit/delete).

Beneficiary Form Fields Include:

  • Application Number (auto or manual), Male Head full details (name, DOB, occupation, address, NIC, special qualifications)
  • Female Head details, Monthly income, Contact number, NIC
  • Children records: studying, dropped out, at university, abroad, other — each with name, DOB, grade/year, gender
  • Special needs, family problems, prior assistance history, photo
  • Status: pending → Active or rejected (with rejection reason)
  • Assessment notes for internal review

Status Workflow:

pending → Active or rejected or deactivate
Bulk Import: Use the Import XLSX button to upload an Excel file of beneficiary records. The system maps columns to the schema automatically. Export the current list with Export XLSX.

7.5 — Subscriptions & Payments

View and manage monthly subscription records across all members. This is a global view (not per-member). For per-member history, use the Member History Modal in the Members module.

  • Filter by member name, month, payment status (Paid / Partial / Unpaid).
  • Add a subscription: select member, enter month, expected amount, payment method, date.
  • Mark partial payments by entering the Paid Amount separately from the expected amount.
  • Advance payments can be flagged with the Is Advance toggle.
  • Export filtered list to CSV.

7.6 — Donations

Track all incoming donations. Two sub-types: External Donations (individual donors) and Family Donations (aid given to registered families).

External Donations:

  • Add donor: Name, Phone, Amount, Donation Type (Zakat / Sadaqah / Fitrana / Lillah / Other), Payment Method, Description, Date.
  • All records appear in the transactions ledger with type = income.

Family Donations:

  • Select a registered family and log aid: Amount, Type (Cash / Dry Ration / Zakat), Description, Date.
  • Tracked per family in family_donations table.

7.7 — Programs & Courses

Manage educational programs run by EDF. Requires courses permission for write operations.

Programs (Courses):

  • Create program: Title, Category (Quran / Skills / Bayan), Target Audience (Men / Women / Children), Max Capacity, Fee, Location, Duration, Description.
  • Assign a Form Template to collect custom application fields.
  • View enrolled applications count per program.

Batches:

  • Each program can have multiple batches (e.g., Morning Batch, Evening Batch).
  • Batch fields: Name, Start Date, End Date, Instructor Name, Max Seats, Location, Status.

Applications:

  • View all applications for a program. Filter by batch and status (Pending / Approved).
  • Approve or reject applications individually.
  • Custom form fields (from the form template) appear alongside standard applicant info.

App Customizer:

  • Build custom application form templates using a drag-and-drop field builder.
  • Supported field types: Text, Number, Date, Select, Checkbox, Textarea.
  • Mark a template as default so all new programs use it automatically.

7.8 — Inventory & Assets

Track physical assets, equipment, and the library book collection.

  • Add Item — Enter: Name, Category, Tracking Type, Serial Number, Quantity, Min Threshold (alert level), Condition, Purchase Price, Warranty Info, QR Code.
  • Library Items — Additional fields: Author, Language, Subject, Library Type (public/restricted).
  • Status Alerts — Items below minimum threshold are highlighted.
  • Search by name or serial number; filter by category and condition.

7.9 — Documents & Files

Centralized file storage for organizational documents.

  • Upload — Click Upload Document, select a file (max 200 MB). Forbidden types: .exe, .sh, .bat.
  • Categories are auto-detected: PDF, Excel, Word, Presentation, Audio, Video, Image, Other.
  • Add an optional description per document.
  • Download — Click the download icon on any document to retrieve the file.
  • Delete — Click the delete icon. File is removed from both the database and the uploads folder.

7.10 — Ramadan Token Distribution

Manages Ramadan aid distribution to beneficiaries via numbered tokens. Requires ramadan permission.

Workflow:

  • 1
    Create a Distribution — Set Year, Title (e.g., "Ramadan 2026 Package"), Collection Date, Location, Voucher Value per token.
  • 2
    Assign Tokens — Bulk-assign tokens to beneficiaries within the distribution. Each beneficiary gets a unique token number.
  • 3
    Mark Collection — On the collection day, scan or select tokens and mark them as Collected. Uncollected entries can be marked Absent.
  • 4
    Review Summary — See counts: Total tokens, Collected, Pending, Absent.
Token Status Flow: pending → collected or absent

7.11 — Volunteers

Registry of volunteers who support EDF activities.

  • Add Volunteer — Name, Phone, Email, Skills (Teaching / Distribution / Admin / Events), Status.
  • Search by name; filter by skill or status.
  • Edit or delete volunteer records.

7.12 — User Accounts (Admin Only)

Visible only to admin role users. Manage staff accounts and their system access.

User List:

  • Shows all users with: Name, Role, Email, Status, and a live online indicator (green dot = active in last 90 seconds).
  • Click a user to expand their permissions and details.

Creating a User:

  • 1
    Click + New User.
  • 2
    Fill in: Username, Full Name, Email, Password, Role.
  • 3
    Assign module permissions using the checkbox grid.
  • 4
    Click Create User. Password is hashed immediately — it is never stored in plain text.

Account Actions:

ActionEffect
Block UserSets user status to blocked. User is immediately rejected on next API call and cookie is cleared.
Unblock UserRestores status to active.
Force LogoutSets force_logout_at timestamp. User's active session becomes invalid immediately, even if their JWT has not expired.
Edit PermissionsToggle any of the 9 module permissions on or off per user.

7.13 — Front Desk (Public Website Management)

Visible to admin and designated staff. Control all content displayed on the public-facing website (index.html, gallery.html, about.html).

Gallery Images:

  • Upload images with a caption, category, display year, and sort order.
  • Toggle image visibility (is_active on/off).
  • Reorder images by changing sort_order values.

Site Content (Page Text):

  • Edit key text blocks on each page (home, about, gallery) using the content editor.
  • Changes are stored in site_content as key-value pairs and rendered by the public pages via the API.

Team Members:

  • Add team member profiles: Name, Role, Bio. An avatar color and letter are auto-assigned.
  • Reorder team members using sort_order.

Partners & Timeline:

  • Partners — List of partner organization names displayed on the about page.
  • Timeline — Historical milestones (year, label, title, description) shown on the about page in chronological order.

Logo Upload:

  • Upload a new site logo via Settings → Logo Upload. The logo is stored in /public/uploads/ and a cache-busting query string is appended automatically.

7.14 — System Logs

Full audit trail of all activity in the system. Accessible from the sidebar.

  • Stats Panel — Total log entries, today's entries, error count, active users, actions breakdown chart.
  • Log Table — Filter by: user, entity (Members/Beneficiaries/etc.), action (CREATE/UPDATE/DELETE), date range.
  • Login Events — Separate tab showing login/logout history with IP addresses and timestamps.
  • Each log entry shows: Who did it, What they did, What was affected (entity + ID), When, and from which IP.
  • Logs can be exported or cleared (admin only).

7.15 — Settings

System-wide configuration. Admin access required for most operations.

Security Measures

  • Password Hashing — All passwords stored as bcrypt hashes. Plain text passwords are never persisted.
  • httpOnly JWT Cookies — Tokens are not accessible to JavaScript, protecting against XSS-based token theft.
  • Role-Based Access Control — admin and super_user bypass permission checks; all other roles require explicit grants.
  • Account Blocking — Blocked users are rejected immediately at the middleware level, not just at login.
  • Force Logout — Remote session invalidation via timestamp comparison, works even on existing valid tokens.
  • IP Logging — All API calls and login events log the client IP for audit purposes.
  • File Upload Restrictions — Executable file types (.exe, .sh, .bat) are blocked on document uploads.
  • XSS Protection — User-controlled strings rendered into innerHTML are sanitized with escapeHtml().
  • Full Audit Trail — Every CRUD operation is logged with the user, entity, action, and timestamp via activityLogger.

File Uploads & Storage

All uploaded files are stored locally in the public/uploads/ directory, which is served as static content. The multer package handles multipart form data.

Beneficiary PhotoInside beneficiary POST/PUT/public/uploads/photos/—Image files
Site LogoPOST /api/settings//public/uploads/—Image files
FaviconPOST /api/settings//public/uploads/—.ico files
Production Note: For production deployments with significant traffic, consider moving file storage to an object storage service (e.g., AWS S3, Cloudflare R2). The current local storage approach is suitable for small-scale deployments.

File Naming

Uploaded files are renamed using a timestamp + random suffix pattern (e.g., 1745000000000-123456789.jpg) to prevent filename collisions and to avoid exposing original filenames in the URL. The original filename is stored in the database as original_name.
